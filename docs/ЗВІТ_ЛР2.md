# Лабораторна робота №2
## Тема: Створення проєкту та базового прототипу гри
### Гра: **Neon Strike** — мультиплеєрний arena FPS у стилі кіберпанк

---

## 1. Обраний ігровий рушій — Godot 4 (C#)

| Критерій | Обґрунтування |
|---|---|
| **Відкритий вихідний код** | Повністю безкоштовний, без роялті та ліцензійних обмежень |
| **Вбудований мережевий стек ENet** | Підтримка peer-to-peer та клієнт-серверного мультиплеєру з мінімальним бойлерплейтом |
| **C# як мова** | Суворе типізування, рефлексія, зручна підтримка в IDE (Rider / VS Code) |
| **3D-рушій першого класу** | CharacterBody3D, RigidBody3D, RayCast3D — всі примітиви FPS-жанру готові до використання |
| **Малий розмір дистрибутиву** | ~100 МБ проти 2+ ГБ у Unity/Unreal, що прискорює ітерацію |
| **Відповідність жанру** | Arena FPS вимагає детермінованої фізики, hitscan-зброї та швидкого циклу смерть/респаун — все вбудовано в рушій |

## 4. Загальна схема взаємодії сцен

```
┌─────────────────────────────────────────────────────────┐
│                     Main.tscn (AutoLoad)                │
│  ┌──────────────┐   HostGame()    ┌────────────────┐    │
│  │  MainMenu    │ ─────────────→  │  Map.tscn      │    │
│  │  (Control)   │   JoinGame(ip)  │  (ігрова арена)│    │
│  └──────────────┘                 └────────────────┘    │
│                                         │               │
│  ┌──────────────┐   UpdateHealth()      ▼               │
│  │   HUD        │ ←──────────── Player.tscn             │
│  │  (CanvasLayer│   UpdateScore()  (CharacterBody3D)     │
│  └──────────────┘   UpdateAmmo()                        │
│                                         │               │
│  ┌──────────────┐   AddFrag()           │               │
│  │ MatchManager │ ←─────────────────────┘               │
│  │  (Node)      │   SubFrag()                           │
│  └──────────────┘   → ClientMatchEnded()                │
│         │                    │                          │
│         └── через 6 с ───→ StartMatch() (перезапуск)    │
└─────────────────────────────────────────────────────────┘
```

**Стани гри:**

```
[MainMenu] ──Host/Join──→ [Gameplay] ──TimeOut──→ [MatchEnd] ──6 s──→ [Gameplay]
                              ↑ Respawn (3 s)          │
                              └────── Death ────────────┘
```

## 7. Базова ігрова логіка

### 7.1 Умови початку гри

| Крок | Опис |
|---|---|
| 1 | Гравець відкриває `MainMenu` і натискає **Host** (створює сервер) або **Join** (вводить IP і підключається) |
| 2 | `Main.StartGame()` приховує меню, показує HUD, завантажує `Map.tscn` |
| 3 | `Map._Ready()` спавнить `Player.tscn` для кожного підключеного peer |
| 4 | `MatchManager.StartMatch()` запускає 15-хвилинний таймер і синхронізує стан через RPC |

### 7.2 Умови перемоги / поразки

| Умова | Опис | Реалізація |
|---|---|---|
| **Перемога** | Найбільша кількість фрагів після закінчення 15 хвилин | `MatchManager._Process` → `ClientMatchEnded` → `HUD.ShowMatchEnd` |
| **Поразка** | Всі інші гравці після визначення переможця | `HUD.ShowMatchEnd` з текстом "PLAYER X WINS" |
| **Смерть** | HP ≤ 0 → `ClientDie()` → приховується меш, вимикається колізія | `HealthComponent.Died` + RPC `ClientDie` |
| **Респаун** | Через 3 секунди після смерті | `_respawnTimer` → `ServerRespawn` RPC → `Respawn()` → нова позиція |

### 7.3 Схема фрагів

```
Вбивство ворога      → +1 фраг (MatchManager.AddFrag)
Смерть від зони/себе → -1 фраг (MatchManager.SubFrag) [може йти в мінус]
Таймер = 0           → MatchManager знаходить max frags → оголошує переможця → через 6 с перезапуск
```

### 7.4 HUD (інформаційний інтерфейс)

| Елемент HUD | Розташування | Дані |
|---|---|---|
| HP-бар з gradient fill | Лівий низ | зелений→жовтий→червоний; ghost drain ефект |
| Лічильник набоїв | Ліво-нижня панель | `current / max`; cyan→orange→red за залишком |
| Назва зброї + слоти | Ліво-нижня панель | підсвітка активного слоту |
| Таймер матчу | Верхній центр | MM:SS; жовтіє при <60 с, червоніє при <20 с |
| Панель фрагів | Правий верх | список всіх гравців; локальний виділений ціаном |
| Hit marker | Центр | червоний на hit, золотий на kill; 0.15 с |
| Damage vignette | Весь екран | червона рамка-шейдер при отриманні пошкоджень |
| Overlay поразки/перемоги | Центр | "YOU WIN!" / "PLAYER X WINS" + рахунок |

## 11. Технічні висновки

### 11.1 Архітектурні рішення

В ході роботи реалізовано **компонентну архітектуру** на основі шаблону «Entity–Component»: логіка руху (`MovementComponent`), логіка здоров'я (`HealthComponent`) та логіка зброї (`WeaponController`) виділені в окремі вузли-компоненти, а не зосереджені в одному монолітному скрипті гравця. Це дозволило:
- незалежно тестувати кожен компонент (зміна параметрів руху не торкалась системи зброї);
- повторно використовувати `HealthComponent` для різних об'єктів — гравця, ящиків (`DestructibleCrate`) та інших NPC без дублювання коду;
- спростити відлагодження: кожен компонент відповідає за строго визначений набір стану.

### 11.2 Мережева синхронізація та авторитет сервера

Реалізовано **клієнт-серверну модель** із сервером як єдиним джерелом істини. Ключові принципи:

| Операція | Хто ініціює | Хто виконує | Механізм |
|---|---|---|---|
| Пошкодження від зброї | Клієнт (власник зброї) | Сервер | `RpcId(1, ServerTakeDamage, ...)` |
| Пошкодження від зони | Сервер (HazardZone) | Сервер | Прямий виклик `ServerEnvironmentDamage()` |
| Синхронізація HP | Сервер | Всі клієнти | `Rpc(ClientSyncHealth, hp)` |
| Смерть | Сервер | Всі клієнти | `Rpc(ClientDie)` |
| Респаун | Клієнт (запит) | Сервер → всі | `RpcId(1, ServerRespawn)` → `Rpc(ClientRespawn)` |
| Фраги | Сервер | Всі клієнти | `Rpc(ClientFragUpdate, peerId, count)` |

Такий підхід відповідає **стандарту авторитативного сервера**, прийнятому в промислових мережевих FPS-іграх (Unreal Engine Replication, Source Engine authoritative server): жоден клієнт не може самостійно змінити ігровий стан іншого гравця.

На практиці виявлено та виправлено специфічну поведінку Godot 4: `RpcId(1, method)` при `CallLocal = false` є **no-op**, коли поточний peer і є сервером (peer ID 1). Це призводило до того, що гравець ніколи не отримував підтвердження респауну і залишався замороженим. Рішення — явна перевірка `Multiplayer.IsServer()` і прямий виклик методу в такому разі.

### 11.3 Фізика та взаємодії

**Hitscan-зброя** (`BasicWeapon`, `LaserSniper`) реалізована через `RayCast3D` з викликом `ForceRaycastUpdate()` кожного кадру стрільби. Це гарантує актуальний результат незалежно від позиції камери між фізичними тіками. Функція `GetAimTarget()` в базовому класі `WeaponBase` трансформує локальну кінцеву точку променя у світовий простір через `GlobalTransform * TargetPosition` — коректний спосіб для довільного напрямку (включно зі стрільбою вгору/вниз), на відміну від `Basis.Z * length`, який дає помилку при нахилі камери.

**Фізичний снаряд** (`Rocket`) використовує `MoveAndCollide()` замість `MoveAndSlide()` для точної реєстрації першого зіткнення без «проковзування» крізь тонкі поверхні. AoE-вибух перевіряє всі тіла в радіусі через `PhysicsDirectSpaceState3D.IntersectSphere()` та застосовує **rocket-jump knockback** — відштовхуючий вектор від центру вибуху, що дозволяє гравцю використовувати ракету для прискорення (класична механіка Arena FPS).

**HazardZone** було переведено на серверну обробку (раніше `TakeDamage` викликався на клієнті, що не синхронізувало смерть). Тепер `_Process` повністю відпрацьовує лише на сервері, що усунуло проблему «гравець стоїть мертвий і ще може стріляти».

### 11.4 Захист від некоректних повторних спрацювань

| Об'єкт | Проблема | Рішення |
|---|---|---|
| HealthPickup / AmmoPickup | Декілька гравців або повторний вхід до підбирання | Прапор `_available = false` одразу при спрацюванні; перевірка на початку `OnBodyEntered` |
| JumpPad | Багаторазовий запуск при утриманні на майданчику | `_cooldownTimer = 0.5 с` — ігнорує нові входи впродовж кулдауну |
| Player смерть | Повторний виклик `ClientDie` при множинних пошкодженнях за один тік | Перевірка `if (_isDead) return` на початку `ClientDie` |
| WeaponController стрільба | Стрільба можлива після смерті (миша захоплена) | Перевірка `if (_player.IsDead) return` у `_Process` |
| HazardZone тік | Пошкодження мертвого гравця | Перевірка `!target._isDead` у `ServerEnvironmentDamage` |

### 11.5 Проблеми, виявлені під час тестування

| Проблема | Причина | Виправлення |
|---|---|---|
| Гравець не респавнився, стояв нерухомо | `RpcId(1)` — no-op на сервері при `CallLocal=false` | Прямий виклик `ServerRespawn()` якщо `Multiplayer.IsServer()` |
| Снаряд летів у хибному напрямку при погляді вгору/вниз | `GetAimTarget()` використовував `Basis.Z * length` (неправильний знак + ігнорував нахил) | Замінено на `GlobalTransform * TargetPosition` |
| AmmoPickup не спрацьовував для зброї крім AR | Пошук зброї за `WeaponIndex` не знаходив збігу при відсутньому `Stats` | Замінено на підбирання набоїв до `controller.ActiveWeapon` |
| Смерть від HazardZone не відбувалась | Пошкодження застосовувалось на клієнті, смерть не синхронізувалась | Повністю серверна обробка через `ServerEnvironmentDamage()` |

### 11.6 Загальний висновок

Прототип продемонстрував, що реалізація мережевого FPS навіть на рівні прототипу вимагає чіткого розмежування клієнтської та серверної логіки на ранньому етапі проєктування. Більшість виявлених помилок були пов'язані саме з порушенням цього розмежування — виклики методів на хибній стороні з'єднання або неврахування поведінки движка при локальному виконанні RPC. Компонентна архітектура та авторитативний сервер забезпечили стабільну основу для подальшого розширення гри.
